# 给其他 AI：如何帮用户改进这套协作规则

若用户把 **`agent-collaboration-kit`** 发给你做建议或改写，请优先遵守下列约束，避免「规则越来越重、互相打架」。

## 请保留的核心约定

- **删除前备份**：[BACKUP_AND_DELETION.md](BACKUP_AND_DELETION.md) 与 `.ai-deletion-backups/` 约定；弱化会破坏用户恢复误删的能力。
- **整夹复制**：`AGENTS.md` 与 `.agents/` 保持同级；用户通过复制整个文件夹接入新项目。
- **项目差异出仓库**：栈、命令、部署路径写在 **`PROJECT_PROFILE.md`（仓库根）**，不要写死在通用协议里（除非是示例占位）。
- **角色名与厂商解耦**：继续使用 **代码 AI / Review AI / 顾问类 AI** 等角色名，不写具体产品名。
- **提交与部署须用户明确授权**：这一条不要弱化。

## 建议的反馈形式

- **补丁式**：指出「哪一节 / 哪个文件」改什么、为何改；避免只给抽象原则。
- **标优先级**：P0（会破坏协作）/ P1（显著增益）/ P2（锦上添花）。
- **标冲突**：若你的建议与现有 `WORKFLOW.md`、`ENGINEERING_NOTES.md` 某条矛盾，请显式写出取舍。

## 内容分层（避免重复）

| 层级 | 适合内容 |
|------|----------|
| `AGENTS.md` | 启动顺序、门禁摘要、指向长文 |
| `.agents/README.md` | 锁、任务、审查流程；短摘要 + 链接 |
| `WORKFLOW.md` | 从需求到发布的阶段与闸门 |
| `ROLES.md` | 角色定义与 STATE 字段 |
| `ENGINEERING_NOTES.md` | 通用工程注意项 |
| `PROJECT_PROFILE.template.md` | 项目类型、命令、严审补充 |

新增大段时：先判断应落在哪一层；**不要把 ENGINEERING 全文搬进 AGENTS**。

## 反例（通用包中应避免）

- 把单个应用（如某编辑器插件）的页面名、目录结构写进默认协议，导致其他项目读困惑。
- 强制引入脚本安装器或在线服务，破坏「只复制文件夹」。

## 版本说明

用户可能 fork 本包；若你输出「推荐合并稿」，可注明适用于 **agent-collaboration-kit 某结构**，并建议用户保留 `README.md` 中的「先读顺序」与 `META_FOR_REVIEWERS` 本文件。
