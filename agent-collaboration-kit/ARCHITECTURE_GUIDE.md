# 架构设计与技术选型指导

> 与具体技术栈弱相关。具体命令、目录、语言版本以 **`PROJECT_PROFILE.md`** 为准。
>
> 本文档解决：接到一个新功能需求后，**如何确定架构方向、如何对比技术路线、如何记录决策**。

## 何时需要正式架构设计

以下情形建议在编码前先输出简要方案（哪怕只有几段话），写入任务卡 `Approach` 或 `docs/design/*.md`：

| 情形 | 最低限度产出 |
|------|-------------|
| 涉及持久化/数据格式变更 | 数据模型变化、迁移策略、回滚方式 |
| 引入新依赖或运行时 | 选型对比（2～3 个候选）、弃用理由、许可证 |
| 跨模块/跨服务调用 | 接口契约、错误传播、版本兼容 |
| 并发/异步/分布式假设 | 竞态边界、超时与重试、幂等策略 |
| 安全敏感（鉴权、支付、隐私） | 威胁面、最小权限、敏感数据流转 |
| 用户交互关键路径（首屏、表单、导航） | 状态机、异常分支、无障碍 |
| 单次改动跨越 `PROJECT_PROFILE` 设定的文件阈值（默认 >3 个核心业务文件） | 影响范围清单、模块依赖图（文字版即可） |

**可跳过正式设计**的情形：单文件文案调整、样式微调、已有模块的纯内部重构（接口不变）、配置值修改。

## 设计前输入清单

动手写方案前，确认已收集：

1. **功能需求**：用户想要什么（用户故事格式：作为…我希望…以便…）
2. **非功能需求**：性能、并发量、离线可用、启动时间、包体积、兼容性
3. **约束**：
   - 技术约束（已有栈、禁止引入的语言/框架）
   - 时间约束（MVP  deadline、能否分阶段交付）
   - 合规约束（数据本地化、无障碍标准、许可证兼容）
4. **反需求（Non-goals）**：明确不做什么，防止范围蔓延
5. **现有架构上下文**：当前模块边界、数据流、已知技术债

## 技术选型框架（四步法）

### Step 1：列出约束与优先级

将约束按**不可协商**和**可权衡**分类：

| 优先级 | 示例 |
|--------|------|
| 硬性约束 | 必须兼容 Node 18；禁止 GPL 依赖；必须离线可用 |
| 高优先级 | 启动 < 1s；包体积 < 500KB；单测覆盖 > 80% |
| 可权衡 | 类型安全 vs. 开发速度；运行时性能 vs. 构建复杂度 |

### Step 2：提出候选方案

通常 2～3 个候选即可，避免分析瘫痪：

- **方案 A**：最小改动（基于现有架构扩展）
- **方案 B**：引入新抽象（新模块、新依赖、新服务）
- **方案 C**：激进重构（仅在技术债严重且用户授权时考虑）

### Step 3：对比与取舍

使用以下维度对比，在任务卡或设计文档中记录：

| 维度 | 方案 A | 方案 B | 方案 C |
|------|--------|--------|--------|
| 实现复杂度 | 低 | 中 | 高 |
| 对现有代码影响 | 小 | 中 | 大 |
| 性能/可扩展性 | 够用 | 好 | 最好 |
| 引入新依赖 | 无 | 1 个 | 多个 |
| 回滚难度 | 低 | 中 | 高 |
| 维护成本 | 低 | 中 | 高 |
| 与长期规划契合度 | 中 | 高 | 高 |

**决策原则**：
- 默认优先方案 A（最小改动），除非有明确证据表明方案 A 会在 3～6 个月内造成显著痛苦
- 新依赖引入须符合 `PROJECT_PROFILE.md` 中的依赖策略
- 涉及数据持久化时，**回滚难度**权重自动提高

### Step 4：记录决策（ADR）

将决策写成简要记录，推荐放在：

- 任务卡 `Approach` 小节（轻量场景）
- `docs/design/<topic>.md` 或 `docs/adr/NNN-title.md`（较重场景）

ADR 最小内容：
- **背景**：要解决什么问题
- **决策**：选了什么
- **取舍**：放弃了什么、为什么
- **影响**：需要改的模块、接口、配置
- **回滚**：如何撤销

## 模块边界划分原则

### 单一职责

- 一个模块/文件只应对付**一个变化原因**
- 拆分信号：如果改 A 功能时总要连带改 B 文件，说明边界可能不清

### 依赖方向

- 核心业务逻辑**不依赖**框架/平台/UI 细节
- 底层工具（日志、网络、存储）可被上层依赖，反之不行
- 绘制依赖图时，箭头应指向**稳定层**（变化少的层）

### 接口先于实现

- 先定义模块对外暴露的接口（函数签名、类型、事件）
- 再写内部实现
- 接口变更成本高，设计时预留适度的扩展点（但不要过度抽象）

## 数据流与状态设计

### 选择状态位置

| 状态位置 | 适用场景 | 注意 |
|----------|----------|------|
| 组件/页面局部状态 | 纯 UI 反馈（展开/折叠、输入中） | 不需要持久化 |
| 模块级状态 / Store | 跨组件共享的业务状态 | 注意订阅泄漏、更新粒度 |
| URL / 路由参数 | 可分享、可刷新的页面状态 | 敏感信息不入 URL |
| 持久化存储 | 用户数据、配置、缓存 | 迁移策略、版本号、加密 |
| 服务端 / 数据库 | 多设备同步、协作数据 | 离线策略、冲突解决 |

### 状态变更原则

- 同一数据有**单一事实来源**（SSOT）
- 状态变更路径可追溯（日志、事件、action 名）
- 避免隐式双向绑定导致的状态同步 bug

## 常见项目类型的架构参考

> 以下为**参考模式**，非强制。具体以 `PROJECT_PROFILE.md` 和项目现有架构为准。

### Web 前端（SPA / 多端应用）

- **关注点**：路由、状态管理、API 层、组件粒度、首屏加载
- **常见分层**：
  - `pages/` 或 `views/` — 页面组合
  - `components/` — 可复用 UI
  - `hooks/` / `composables/` — 可复用逻辑
  - `services/` / `api/` — 网络请求与数据转换
  - `stores/` — 全局状态
  - `utils/` — 纯函数工具（无业务逻辑）
- **决策点**：客户端状态 vs. 服务端状态比例、缓存策略、按需加载

### CLI / 工具脚本

- **关注点**：参数解析、配置加载、错误码、输出格式（TTY vs. 管道）、退出码
- **常见分层**：
  - `commands/` — 子命令实现
  - `config/` — 配置读取与验证
  - `core/` — 纯业务逻辑（可测试）
  - `ui/` — 输出格式化（进度条、表格、颜色）
- **决策点**：配置文件格式、插件机制、是否守护进程化

### Library / SDK

- **关注点**：API 表面大小、版本兼容、Tree-shaking、文档、类型导出
- **常见分层**：
  - `src/` — 核心实现
  - `types/` — 公开类型定义
  - `adapters/` — 平台适配（Node / Browser / Deno）
- **决策点**： peer dependency 策略、semver 规则、-breaking change 节奏

### Mobile / 桌面应用

- **关注点**：生命周期、离线、平台 API、发布渠道、包体积
- **决策点**：原生 vs. 跨平台、本地数据库选型、推送/通知策略

### 后端 API / 服务

- **关注点**：路由、鉴权、验证、错误处理、日志、限流、部署
- **常见分层**：
  - `handlers/` / `controllers/` — HTTP 入口
  - `services/` — 业务逻辑
  - `repositories/` / `models/` — 数据访问
  - `middleware/` — 横切关注点（日志、鉴权、限流）
- **决策点**：REST vs. RPC、认证方案、数据库选型、缓存层

## 技术债管理

- **识别**：代码中标记 `TODO`、`FIXME`、`HACK`，并在任务卡或 `docs/tech-debt.md` 中汇总
- **评估**：每条技术债标注「不还的后果」和「还的成本」
- **偿还时机**：
  - 低成本的债：顺手还（改附近代码时）
  - 高成本但高风险的债：排入独立任务卡，经用户确认后做
  - 高成本低风险的债：记录即可，不主动重构
