# 删除前备份与恢复（AI 与人类共用）

**原则**：任何**不可逆或难以从 git 恢复**的删除、覆盖、重命名前，先写入备份区，并保留一段固定时间，便于误删时手工恢复。

## 适用范围

- 使用 `rm`、`delete_file`、空目录清理、批量重构导致的文件消失  
- 覆盖写入重要文件前（若无法用 `git checkout` 立即找回未提交内容）  
- **已跟踪文件**：优先用 `git`（见下）；未跟踪、生成物、或用户明确要求物理删除时，用备份区  

## 备份区位置（默认）

- **Git 仓库根目录**下：`.ai-deletion-backups/`
- 须在 **`.gitignore`** 中忽略该目录（避免把备份当代码提交）。  
- 若项目在 `PROJECT_PROFILE.md` 中另有约定，从其约定。

## 目录结构

```text
.ai-deletion-backups/
└── 2026-05-11_1430_rename-review-templates/    # 建议：日期_时间_简短英文slug
    ├── MANIFEST.txt                             # 必填：谁、为何、原路径列表
    └── files/                                   # 与原路径镜像或平铺，二选一写清
        └── ...
```

**`MANIFEST.txt` 至少包含**：

- 备份时间（UTC 或本地时区，写清）  
- 操作者：`code-ai` / `review-ai` / `human`  
- 任务卡或会话摘要（一句）  
- 每一行一个：**原始相对路径**（相对仓库根）  

## 保留时间

- **默认保留 30 天**；到期后由维护者删除整段子目录。  
- 用户可在 `PROJECT_PROFILE.md` 中改为 7 / 60 / 90 天等，并注明谁负责清理。

## 恢复方式

1. 在 `.ai-deletion-backups/<某次备份>/` 找到对应文件。  
2. 按 `MANIFEST.txt` 将文件拷回仓库内原路径（或用户指定新路径）。  
3. 若曾用 git 删除已提交文件，也可用 `git show <commit>:path` 恢复；备份区作为**未提交或混乱工作区**时的兜底。

## 已跟踪文件的首选路径

- 若删除仅为了「整理」，优先：**`git rm` 前另开分支**或**只做 `git mv` 到 `_archive/`** 仍保留历史，而不是物理删。  
- 备份区作为 **git 之外** 的保险，不替代版本控制。

## Review AI

- 审查时若发现实现方删除文件却**无备份、无 git 历史可查**，可列为 **P1**：要求补备份或说明恢复路径。
