# 通用工程注意事项（与栈弱相关）

具体命令、目录、语言版本以 **`PROJECT_PROFILE.md`** 为准。下列适用于多数软件项目；**Review AI** 与 **代码 AI** 在改动相关区域时应主动核对。

---

## 安全与合规

- **密钥与令牌**：勿写入仓库；使用环境变量或密钥管理；在 `PROJECT_PROFILE` 写明禁止提交的路径（如 `.env`）。
- **注入类风险**：SQL/OS 命令/模板/HTML 拼接时优先参数化或框架推荐 API。
- **依赖**：新增依赖前确认许可证与用户授权；注意供应链（锁定版本、审计工具若项目已有）。
- **敏感数据日志**：避免打印密码、完整 token、个人身份信息。
- **输出编码**：向 HTML/JSON/终端输出用户可控内容时，确认转义或框架自动转义。
- **最小权限**：服务账户、API Key、文件权限按最小必要授予。

---

## 错误处理与可观测性

- **用户可见错误信息应可理解且可操作**；内部细节用日志级别区分。
- **关键路径应有可诊断信息**（请求 id、关键状态）；具体栈按项目现有实践（日志库、tracing）。
- **错误传播**：
  - 底层错误应包装为有业务语义的错误，避免把原始堆栈直接抛给用户
  - 异步错误（Promise、回调、事件）必须有 `.catch` / `try-catch` / 错误事件处理
  - 不要吞掉错误（空的 catch 块），至少记录日志
- **日志级别约定**：
  - `DEBUG`：开发调试信息
  - `INFO`：正常业务流程节点
  - `WARN`：可恢复异常、降级行为
  - `ERROR`：需要人工介入的故障

---

## 兼容性与版本

- 在 `PROJECT_PROFILE` 声明：**语言/运行时最低版本**、目标浏览器/平台。
- 行为变更若影响旧数据或旧客户端，在任务卡中写明**迁移或降级策略**。
- **API 版本**：若项目暴露公开 API，修改接口时考虑向后兼容（加字段不改旧字段、新端点 vs. 改旧端点）。
- **数据格式版本**：持久化数据（文件、数据库、localStorage）应带版本号，读取时做降级或迁移。

---

## 性能与资源

- 对热路径（高频调用、大数据量）避免无必要的分配与同步阻塞；**先测量再优化**。
- 长时间任务：考虑超时、取消、进度反馈（按项目形态）。
- **懒加载**：首屏/启动无关的代码、数据、资源应延迟加载。
- **缓存策略**：
  - 区分「永远不变」「可变但可过期」「必须实时」的数据
  - 缓存失效应有明确机制（TTL、事件驱动、版本戳）
- **大数据量**：
  - 列表/表格考虑分页、虚拟滚动、流式处理
  - 避免一次性加载全部数据到内存

---

## 可访问性与国际化（若适用）

- **UI**：键盘可操作、焦点管理、对比度、语义标签（按项目标准）。
- **文案**：若项目支持多语言，勿硬编码不可翻译串（按现有 i18n 方案）。
- **无障碍最低要求**：
  - 交互元素有可聚焦的语义标签
  - 颜色不是唯一信息载体
  - 表单有明确的错误提示与关联标签

---

## 并发与异步

- **共享可变状态**：多线程/多协程/事件循环中共享状态须加锁、原子操作或避免共享。
- **竞态条件**：异步操作完成顺序不确定时，确保后续逻辑不依赖未完成的副作用。
- **取消与清理**：
  - 异步任务应支持取消（AbortController、context cancel、disposable）
  - 组件/页面卸载时取消未完成的请求、清理定时器、移除事件监听
- **死锁与活锁**：持有多个锁时保持固定获取顺序；避免在锁内调用外部回调。

---

## 数据与存储

- **数据库/文件操作**：
  - 写操作先验证再执行（校验规则在应用层和存储层双重保护）
  - 批量操作有上限，防止内存或连接耗尽
  - 事务边界与回滚策略明确
- **序列化/反序列化**：
  - 不信任外部输入，反序列化前校验结构和类型
  - 使用 JSON 时警惕 `__proto__`、`constructor` 等原型污染
- **迁移**：
  - 数据迁移须有向下兼容的读取逻辑（先读旧格式，再迁移）
  - 大规模迁移应分批、可回滚、有进度反馈

---

## 删除与备份

- **删除或覆盖前**：须按 [BACKUP_AND_DELETION.md](BACKUP_AND_DELETION.md) 写入 `.ai-deletion-backups/`（或 `PROJECT_PROFILE` 约定路径），并写 `MANIFEST.txt`；**默认保留约 30 天**。已跟踪文件优先用 git 分支 / `git mv` 到归档，备份区作兜底。
- **Review AI** 可对「无备份的破坏性删除」提 P1。

---

## 代码与协作纪律

- **最小改动**：只改完成任务所需的文件与行数；避免顺带大重构。
- **风格一致**：命名、目录结构、错误处理方式与仓库现有代码一致。
- **生成物**：若仓库包含构建输出，须与源文件**同轮**更新；禁止手改生成物的规则写在 `PROJECT_PROFILE`。
- **换行与 diff**：异常巨大的格式化 diff 先检查 `git ls-files --eol` 或项目格式化策略。
- **不要留下调试代码**：提交前移除 `console.log`、临时注释、死代码。
- **配置与代码分离**：环境相关值（URL、超时、开关）应走配置，不要硬编码。

---

## API 与接口设计

- **命名**：函数/端点名应体现动作 + 对象（如 `createUser` 而非 `userCreate` 或 `handleUser`）
- **参数**：参数过多（>4 个）时改用选项对象；布尔参数考虑是否应拆成两个语义函数
- **返回值**：成功/失败的状态应明确；不要返回 `null` 和空数组混用表示「无结果」（团队内统一即可）
- **错误码**：若项目有错误码体系，新增错误须按规则分配，不随意复用

---

## 测试

- 运行 `PROJECT_PROFILE` 中列出的测试/类型检查；新逻辑尽量有对应测试或手动验收步骤（小项目可仅后者，但须在任务卡写明）。
- **测试数据**：使用工厂函数或 faker 生成，不要依赖生产数据。
- **外部依赖**：测试中的网络、数据库、文件系统应 mock 或使用内存/测试替身。
- ** flaky 测试**：遇到偶发失败的测试，先标记跳过并建任务卡修复，不要反复重试掩盖问题。

---

## 提交与发布前

- 与安全、迁移、密钥相关的改动：对照 [.agents/reviews/PRECOMMIT_CHECKLIST.md](.agents/reviews/PRECOMMIT_CHECKLIST.md) 与严审规则。
- **提交信息**：
  - 使用 `[code-ai]` / `[review-ai]` 前缀
  - 标题一行概括改了什么（不是「修复 bug」而是「修复用户列表在空数据时崩溃」）
  - 正文说明「为什么」和「影响范围」，关联任务卡编号（若有）
