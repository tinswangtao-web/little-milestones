# 代码质量与测试标准

> 与具体技术栈弱相关。具体命令、目录、语言版本以 **`PROJECT_PROFILE.md`** 为准。
>
> 本文档提供：**Review AI 的系统检查框架**、**代码 AI 的自查清单**、以及**测试策略指导**。

## 质量维度总览

| 维度 | 含义 | 检查时机 |
|------|------|----------|
| **正确性** | 代码是否按需求工作，边界条件是否覆盖 | 实现后、Review 时 |
| **可读性** | 命名是否表意、结构是否清晰、注释是否必要 | Review 时 |
| **可维护性** | 耦合度、重复、模块边界、技术债 | Review 时 |
| **可测试性** | 纯逻辑是否可独立测试、副作用是否可控 | 设计时、Review 时 |
| **性能** | 热路径是否有明显浪费、大 O 是否合理 | 实现后（必要时测量） |
| **安全性** | 注入、XSS、越权、敏感信息泄露 | 设计时、Review 时 |
| **兼容性** | 是否破坏旧数据、旧 API、旧客户端 | 设计时、Review 时 |

## Review AI 检查框架

审查时代码 AI 的产出时，按以下维度系统扫描，避免遗漏：

### 1. 需求对齐（Functional Correctness）

- [ ] 本轮目标是否全部达成？有无遗漏的验收标准？
- [ ] 边界条件是否处理（空值、越界、空数组、网络断开、权限不足）？
- [ ] 错误路径是否有用户可理解的反馈？
- [ ] 是否引入了未在任务卡中声明的新行为？

### 2. 代码结构（Structure & Maintainability）

- [ ] 函数/类/模块是否单一职责？
- [ ] 是否有明显重复可提取？
- [ ] 循环依赖或隐式全局状态？
- [ ] 新接口与现有接口是否一致（命名风格、错误处理方式）？
- [ ] 是否破坏了已有抽象层级（如 UI 层直接调数据库）？

### 3. 正确性与健壮性（Robustness）

- [ ] 异步操作：是否有未处理的 Promise 拒绝/异常？
- [ ] 资源管理：文件、连接、定时器、事件监听是否正确释放？
- [ ] 并发：共享可变状态是否有竞态？
- [ ] 类型：是否有 any/unknown 的滥用？类型收窄是否完整？
- [ ] 数字/日期/时区：计算是否正确？精度是否足够？

### 4. 安全（Security）

- [ ] 用户输入是否经过校验/转义？
- [ ] 是否有拼接字符串生成命令/SQL/HTML？
- [ ] 敏感数据（token、密码）是否出现在日志或错误信息中？
- [ ] 是否有越权访问路径（未鉴权可触发的敏感操作）？
- [ ] 依赖包是否有已知漏洞？

### 5. 性能与资源（Performance）

- [ ] 热路径是否有不必要的循环、分配或同步阻塞？
- [ ] 大数据量场景：是否考虑了分页/流式/虚拟化？
- [ ] 是否有内存泄漏嫌疑（闭包持有大对象、未清理的订阅）？
- [ ] 引入的新依赖体积是否合理？

### 6. 测试与验证（Testability）

- [ ] 新逻辑是否有可测试的入口（纯函数、可 mock 的接口）？
- [ ] 复杂条件分支是否有对应测试或手工验收步骤？
- [ ] 构建/类型检查/ lint 是否通过？
- [ ] 是否破坏了已有测试？

### 7. 文档与注释（Documentation）

- [ ] 公开 API 是否有文档/类型注释？
- [ ] 非直观逻辑是否有注释说明「为什么」而非「做什么」？
- [ ] 配置变更是否在文档/示例中同步？

## 代码 AI 自查清单（实现后自测前）

在交给 Review AI 前，代码 AI 应对照以下快速清单自检：

```markdown
- [ ] 我是否只改了完成任务所需的文件与行？（最小改动）
- [ ] 我是否运行了 PROJECT_PROFILE 中指定的构建/测试命令？
- [ ] 新增函数/模块是否有自解释的命名？（不需要注释也能读懂）
- [ ] 边界条件（null、空数组、最大值、异常）是否已处理？
- [ ] 是否有未处理的 Promise/异步异常？
- [ ] 资源（文件句柄、事件监听、定时器）是否正确清理？
- [ ] 是否引入了未声明的依赖？
- [ ] 生成物（如编译输出、CSS）是否与源码同轮更新？
- [ ] 用户可见的文案/错误信息是否自然、无错别字？
- [ ] 若涉及 UI：键盘可操作、焦点管理、移动端适配是否考虑？
```

## 测试策略

### 测试分层

| 层级 | 范围 | 速度 | 何时写 | 工具举例 |
|------|------|------|--------|----------|
| **单元测试** | 单个函数/类，无外部依赖 | 毫秒级 | 写逻辑时或紧随其后 | Jest, Vitest, pytest, Go test |
| **集成测试** | 模块间交互，含真实/内存级依赖 | 秒级 | 接口稳定后 | Supertest, Playwright API, Testcontainers |
| **E2E 测试** | 完整用户流程 | 分钟级 | 核心流程稳定后 | Playwright, Cypress, Selenium |
| **手工验收** | 真实环境、真实用户视角 | 人速 | 每轮交付前 | 人 |

### 测试原则

1. **优先测边界与错误路径**：正常路径往往容易对，边界容易错
2. **纯逻辑必测**：无外部依赖的计算函数必须有单元测试
3. **UI 逻辑与渲染分离**：业务逻辑抽成纯函数测；UI 层用 E2E 或视觉回归
4. **不要测实现细节**：测行为（输入→输出），不要测内部调用顺序（除非那是契约）
5. **测试也是代码**：保持可读性，命名应说明「在什么条件下期望什么结果」

### 小项目的务实做法

若项目暂无测试框架：

- 在任务卡 `Verification` 中写明**手工验收步骤**（逐步操作 + 期望结果）
- 纯逻辑函数写简单的「自运行脚本」放在 `scripts/smoke-test.js` 或同级位置
- 当项目规模增长后，再引入正式测试框架

## 重构指导

### 何时重构

| 触发条件 | 处理方式 |
|----------|----------|
| 改一个功能需要 touching 多个不相关文件 | 考虑提取中间层 |
| 同一逻辑复制粘贴超过 2 次 | 提取公共函数/组件 |
| 函数超过 50 行或嵌套超过 3 层 | 拆分子函数 |
| 命名与实现不符 | 先改名，再改实现 |
| 现有测试频繁因无关改动而失败 | 测试耦合了实现细节，需重写测试 |

### 重构纪律

- **重构与功能修改分开提交**：不要边改功能边重构，增加审查难度
- **有测试罩着再重构**：先补测试（或确认已有测试覆盖），再动结构
- **小步前进**：一次重构只做一个变换（提取函数 → 改名 → 搬移），每步可验证
- **用户授权**：大范围重构（影响 >5 个文件或改变公开接口）须用户确认，单独建任务卡

## 技术债登记格式

在代码或任务卡中记录技术债时，使用统一格式便于后续追踪：

```markdown
## Tech Debt

- [ ] **ID**: T001 | **位置**: `src/services/user.ts:45`
  - **问题**: 用户权限检查分散在 3 个文件，重复逻辑
  - **后果**: 改权限规则时容易漏改，已出现一次 bug
  - **建议**: 提取 `PermissionService`，统一入口
  - **成本**: 低（约 2 小时）| **优先级**: P1
```

## 与 ENGINEERING_NOTES 的关系

- **ENGINEERING_NOTES.md**：安全、依赖、兼容、删除备份等**纪律性**要求
- **QUALITY_STANDARDS.md**：代码结构、测试、重构等**质量实践**
- 两者在 Review 时都应被对照
