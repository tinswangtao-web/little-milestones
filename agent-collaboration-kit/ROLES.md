# 多 AI 角色模型

角色名称**不绑定**具体模型或产品：同一底层服务在不同会话中可扮演不同角色。以 **`.agents/STATE.md`** 的 `owner`、`awaiting` 为准记录当前回合谁在做什么。

## 标准角色

| 角色 | 职责 | 默认写业务代码？ |
|------|------|------------------|
| **代码 AI** | 实现、构建、自测、按授权的提交与部署；维护任务卡与交接卡 | 是（默认唯一） |
| **Review AI** | 审查 diff/commit；结论与风险分级；最小复测建议 | 否（除非用户授权且写入 `STATE.md`） |
| **顾问类 AI** | 产品、架构、UX、流程建议；帮澄清需求与方案 | 否 |
| **文档 / Prompt 撰写**（可选） | 仅文档、注释规范、Prompt 资产；不写业务逻辑时可单独标为 `owner` | 仅声明的文档范围 |

## 顾问的细分（非必须单独列名）

在任务卡或会话说明中可标注顾问侧重：

- **产品**：优先级、用户故事、验收口径  
- **架构**：模块边界、接口、技术选型对比  
- **UX**：交互与文案（与实现协作）

顾问的产出**须落到任务卡（或设计文档）**后再由 **代码 AI** 动代码；避免「口头方案未记录就改仓库」。

## 与 STATE.md 的对应关系（建议字段）

- **`owner`**：当前对 **`write-scope`** 负责的一方，通常为 `code-ai`；文档任务可为 `docs-ai` 或仍用 `code-ai` 但 scope 仅限文档。
- **`awaiting`**：如 `user`、`review-ai`、`code-ai`，表示卡在等待谁。
- **`last-actor`**：上一回合动作方，便于日志与审计。

具体取值可为自由文本，团队内统一即可；关键是**同一时刻对同一 `write-scope` 只有一个 owner**。

## 并行与冲突

- 仅当 **`write-scope` 不重叠** 且不依赖未完成修改时，允许多方并行（例如一人改 `docs/**`，一人改 `src/**` 中无交集文件）。
- **同一文件**不得被两个 owner 同时持有写锁；见 `LOCK.md`。

## 用户

用户始终拥有：需求定义、验收结论、是否 **commit**、是否 **部署** 的最终决策权。
