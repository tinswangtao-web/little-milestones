# 从需求到发布：全流程

本文件与具体技术栈无关；细节命令与目录以仓库根的 **`PROJECT_PROFILE.md`** 为准。

## 阶段说明

### 1. 需求澄清（Discovery）

- **目标**：把「想要什么」写成可验证的描述，减少实现中途返工。
- **主导**：用户 + 任意 **顾问类 AI**（产品/领域）；**代码 AI** 可参与拆问与边界确认。
- **产出**（写入任务卡或临时笔记）：
  - 问题陈述、用户场景、**验收标准**（可勾选）
  - **非目标**（明确不做的事）
  - 约束：时间、兼容环境、合规、是否允许新依赖等（与 `PROJECT_PROFILE.md` 对齐）

#### 需求拆解模板

将模糊需求转化为可执行项：

```markdown
## 用户原始需求
（用户原话，一字不改）

## 用户故事
作为 [角色]，我希望 [动作]，以便 [价值]

## 验收标准（必须可验证）
- [ ] 给定 [前置条件]，当 [动作]，那么 [可观察结果]
- [ ] ...

## 非目标（明确不做）
- 不做 ...
- 不支持 ...

## 约束与假设
- 时间：...
- 兼容性：...
- 技术约束：...
```

#### 需求澄清检查清单

- [ ] 用户描述的「问题」和「期望的解决方案」是否区分清楚？（有时用户给的是方案而非问题）
- [ ] 是否存在多种用户角色？各自的需求是否冲突？
- [ ] 正常路径之外的边界场景是否已识别？（空数据、网络失败、权限不足、极大/极小值）
- [ ] 是否与现有功能存在交互或冲突？
- [ ] 是否有视觉/交互参考（截图、草图、竞品）？

### 2. 方案与架构（Design）

- **目标**：选定实现路径、模块边界、关键技术点；大需求避免「直接写代码再想」。
- **主导**：**代码 AI** 或 **顾问类 AI（架构向）**；复杂时由用户决定听哪一方为主。
- **产出**：
  - 任务卡中 `approach` 小节：可选方案、**取舍理由**、影响范围
  - 可选：仓库内 `docs/design/<topic>.md` 或 ADR 式短文（不强制目录名）
- **何时可跳过**：
  - 任务卡注明 **轻量 / 无独立设计阶段**（例如单文件小改、文案调整）
- **何时禁止跳过**（须先有简要方案再动代码）：
  - `PROJECT_PROFILE.md` 中严审触发条件（存储/迁移/安全/鉴权等）；**默认可作为起点的典型项**（项目可删减或加重）：
    - 持久化、数据迁移、序列化格式变更
    - 鉴权、安全敏感逻辑
    - 并发、事务、幂等等分布式假设
    - 移动端/桌面关键交互路径（键盘、触摸、导航栈、窗口）
    - 单次改动跨越超过 **3** 个核心业务文件（阈值可在 `PROJECT_PROFILE` 调整）
  - 单次改动跨多模块或引入新依赖/新运行时
  - 用户明确要求「先方案后代码」

#### 方案设计模板

```markdown
## Approach

### 候选方案

#### 方案 A：[名称]（最小改动）
- 做法：...
- 优点：...
- 缺点：...

#### 方案 B：[名称]（引入新抽象）
- 做法：...
- 优点：...
- 缺点：...

### 决策
选择 **方案 X**，理由：...

### 影响范围
- 修改模块：...
- 新增接口：...
- 数据变化：...
- 回滚方式：...
```

详细架构指导见 **[ARCHITECTURE_GUIDE.md](ARCHITECTURE_GUIDE.md)**。

### 3. 任务卡与范围（Task）

- **目标**：把设计和需求固化成 `.agents/tasks/*.md`，并声明 `write-scope`、阶段、回滚说明（若适用）。
- **主导**：**代码 AI**（创建/更新任务卡）；用户确认优先级与验收。
- **产出**：见 [.agents/tasks/_template.md](.agents/tasks/_template.md)。

#### 推荐状态流转

- 任务卡与 `STATE.md` 的 `Current Task.status` 推荐共用一套状态：
  - `planned`
  - `in-progress`
  - `awaiting-review`
  - `awaiting-user`
  - `awaiting-deploy`
  - `done`
  - `cancelled`
- `STATE.md` 额外允许 `idle`，表示当前没有活跃任务。
- `awaiting` 字段单独表示“在等谁”，不要把等待对象混进 `status`。

#### 任务拆分原则

| 情形 | 拆分方式 |
|------|----------|
| 一个需求涉及多个独立页面/模块 | 每个页面/模块一张任务卡 |
| 有前端 + 后端 + 数据变更 | 按层拆卡，标注依赖关系 |
| 有「先调研再实现」的两阶段 | 调研一张卡，实现一张卡 |
| 实验性改动 | 单独一张卡，含回滚说明 |
| 纯重构（无功能变化） | 单独一张卡，不混在功能卡里 |

### 4. 实现与自测（Implement）

- **目标**：最小合理改动；通过 `PROJECT_PROFILE` 中的构建/测试命令自检。
- **主导**：**代码 AI**（占 `LOCK.md`）。
- **产出**：代码与配置变更；`log.md` / `STATE.md` 更新；交接卡草稿。

#### 实现纪律

1. **先验证环境**：运行 `PROJECT_PROFILE` 指定的构建命令，确认基线通过
2. **小步提交（到工作区，非 git commit）**：每完成一个子功能自测一下，不要一次性改 20 个文件再测
3. **边写边测**：新函数随手验证边界条件；UI 改动随时预览
4. **保持 diff 可读**：无关的格式化变动另起任务卡或避免混入
5. **中间产物清理**：临时日志、调试代码、实验文件在提交前移除

详细质量与测试指导见 **[QUALITY_STANDARDS.md](QUALITY_STANDARDS.md)**。

---

### 5. 审查（Review）

- **目标**：第二双眼睛查缺陷、风险、遗漏；**不**默认重写业务代码。
- **主导**：**Review AI**。
- **产出**：`.agents/reviews/` 下审查文或按 [REVIEW_OUTPUT_TEMPLATE.md](.agents/reviews/REVIEW_OUTPUT_TEMPLATE.md) 输出。

#### Review 输入

Review AI 应获取以下信息：
1. `IMPLEMENTATION_REVIEW_HANDOFF.md`（交接卡）
2. `git diff`（或 commit range）
3. 当前任务卡
4. `PROJECT_PROFILE.md`（了解项目约束）

#### Review 维度

系统检查框架见 **[QUALITY_STANDARDS.md](QUALITY_STANDARDS.md)** 的「Review AI 检查框架」。

---

### 6. 用户验收与发布（Ship）

- **目标**：用户按验收标准确认；仅在用户明确授权后 **commit** 与 **部署/同步**（若项目有第二步）。
- **主导**：用户；**代码 AI** 执行授权范围内的 git/部署操作。

#### 验收流程

1. **代码 AI** 提供「建议用户验收步骤」（写在交接卡中）
2. **用户**按步骤操作，对照任务卡中的验收标准勾选
3. **用户**给出明确结论：「通过 / 需要修改 / 放弃」
4. **通过后等待用户决定** → 任务通常标为 `awaiting-user`
5. **项目有部署/同步步骤** → 等待该动作时可标为 `awaiting-deploy`
6. **用户明确授权 commit / deploy 并完成收尾** → 更新 `STATE.md` 为 `done` 或切到下一任务

## 产出物放哪里（约定）

| 内容 | 推荐位置 |
|------|----------|
| 长期栈、命令、严审规则 | `PROJECT_PROFILE.md`（仓库根） |
| 当前需求、方案摘要、验收 | 任务卡 `.agents/tasks/*.md` |
| 较长设计说明 | `docs/design/` 或项目自有文档目录（在 `PROJECT_PROFILE` 声明） |
| 协作状态 | `.agents/STATE.md`、`LOCK.md`、`log.md` |
| 实现与审查交接 | `IMPLEMENTATION_REVIEW_HANDOFF.md`、审查输出文件 |
| 技术债汇总 | `docs/tech-debt.md` 或任务卡中 Tech Debt 小节 |

---

## 与非代码工作（文档 / Prompt）

若任务仅为文档、提示词、配置说明而无业务代码：

- 仍使用任务卡与锁，**`write-scope`** 指向对应路径（如 `docs/**`）。
- **Review AI** 同样适用；严审规则可按 `PROJECT_PROFILE` 缩减。

## 迭代与反馈循环

### 小步快跑模式

适合探索性需求或用户也不确定最终形态的场景：

1. **MVP 卡**：最小可用版本，明确标注「实验性」
2. **快速实现**：1～2 轮内出可体验版本
3. **用户试用**：用户实际操作，收集反馈
4. **反馈转新卡**：每个改进点一张新任务卡，不混在旧卡里
5. **旧卡归档**：MVP 卡标记 `done`，后续迭代由新卡承接

### 需求变更处理

- 实现中途用户提出新想法 → **不直接改当前任务卡**，新建任务卡记录
- 若新想法使当前卡部分作废 → 当前卡更新 `Non-goals` 并说明原因，指向新卡
- 避免「任务卡无限膨胀」
